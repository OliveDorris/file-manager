# 项目规则

每次新任务或较大的需求变更开始前，必须先读取本文件 `AGENTS.md` 和 `PROJECT_CONTEXT.md`。如果发现文档存在空白、过期或与实际情况不一致，应先适当补充或修正，再继续执行任务。

每次需求变更要在原有代码基础上进行修改。不要进行大的框架改变，除非用户明确要求。

## 项目名称

文件管理系统

## 技术栈

前端：
Jinja2 服务端模板 + HTML + CSS

后端：
Python 3.12 + FastAPI + Uvicorn

数据库：
SQLite，本地文件数据库，生产路径为 `/home/file-manager/data/file_manager.sqlite3`

## 后端目录结构

当前保持轻量级 Python + FastAPI 架构，生产启动入口仍是根目录 `app.py`，systemd 命令仍使用 `uvicorn app:app`。

新增或调整业务时优先采用轻量分层：

- `app.py`：FastAPI 应用入口、路由注册、认证依赖、少量历史兼容逻辑。
- `repositories/`：数据库查询和持久化逻辑。
- `services/`：业务判断、文件预览、文件权限与审批等业务逻辑。
- `templates/`：Jinja2 页面模板。
- `static/`：CSS 和静态资源。
- `tests/`：自动化测试。

不要继续把新的大业务直接堆到 `app.py`。历史代码中已有的单文件逻辑可以逐步迁移，禁止一次性大范围重构影响生产稳定性。

## 开发规范

1. 严格禁止：Controller / 路由层直接操作数据库。新增或重构代码时应优先使用 service / repository 分层。
2. 错误处理禁止空捕获或吞异常，例如：

```python
try:
    ...
except Exception:
    pass
```

必须记录日志，并返回统一错误。

3. 修改代码前先分析影响范围。
4. 不允许删除已有功能。
5. 禁止直接大范围重构，除非用户明确要求。
6. 当前历史代码中 `app.py` 仍存在路由层直接操作数据库的情况。后续修改应在不破坏功能的前提下逐步改善，不要为了满足分层规则一次性大改框架。

## 日志规范

重要操作必须记录：登录、修改权限、删除数据、审批操作。

日志必须包含：用户、时间、IP、操作。

## AI 协作规范

Codex 修改代码前必须：

1. 分析当前代码。
2. 提出方案。
3. 等待用户确认。

禁止：直接大范围重构。
